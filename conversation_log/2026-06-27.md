| 方案 | 思路 | 真实感 | 复杂度 | 风险 |
|---|---|---|---|---|
| **A. globe.gl CDN** | `<script src="unpkg.com/globe.gl">` 一行调真 3D | ★★★★★ | 低 | 评委网络可能挂 |
| B. three.js CDN 手写球体 | 自己写 mesh + 纹理 | ★★★★★ | 中 | 代码 200+ 行 |
| C. 精修 CSS 伪 3D | SVG 世界地图 + cloud + perspective | ★★ | 中 | 天花板低 |
| D. 内嵌 globe.gl 全部代码 inline | 完全单文件 | ★★★★★ | 高 | 体积爆炸 1MB+ |

**推荐方案 A**。主项目就是 react-globe.gl，CDN 版本效果 1:1。CDN 网络风险可以靠 jsdelivr 镜像兜底。

## 这次 bug 根因（一句话）

`usePreparationProgressStore((s) => s.getWishProgress(...))` 这套写法踩了 React 19 useSyncExternalStore 雷——`getWishProgress` 每次都返回**新对象**，selector 比较时引用永远不相等 → 无限循环警告。PROJECT_LOG Day 8 记过同款（`s.data?.wishes ?? []` 每次新建数组）。

## 修法

不再用 selector 调 `getWishProgress`，改成：
1. 订阅**稳定引用**的 `done` map（`usePreparationProgressStore((s) => s.done)`）
2. 组件内 `useMemo` 缓存计算结果，依赖 `[wish.id, wish.preparation, done]`
3. 只有 `done` 真的变了才重算

这是 zustand + React 19 的标准写法——任何返回新对象的 selector 都会踩这坑，记到 memory。
